# 영차 POC 구현·검수 정의서

> 기준: 현재 리포지토리의 프론트엔드 코드/화면에서 **실제 확인된 내용만** 반영. 추측 기능 없음.
> 대상 폴더: `yc_mobileapp/`(고객 앱), `yc_pc_admin/`(관리자 사이트). **코드 수정 없이 화면·코드 분석만으로 작성.**

---

## 0. 개발 착수 요약

> 이 문서에서 개발자가 반드시 읽어야 할 부분. 이 한 화면만으로 1차 POC 범위가 확정됩니다.

### 서비스 정책 (대표이사 확정 · 불변)
> **영차는 베이 예약 서비스가 아니다.** 고객은 **현재 사용 가능한 베이를 확인하고 즉시 결제한 뒤 바로 이용**한다. 미래 시간이나 특정 시간대의 베이를 미리 예약하는 기능은 **제공하지 않는다.** 예약 등록·변경·취소·대기 및 예약 상태는 개발하지 않으며, 관련 UI·상태·데이터·API를 모두 제거 대상으로 한다. (POC 제외가 아니라 **서비스에서 제공하지 않는 기능**)

### 서비스 핵심 흐름 (통일)
> 사용 가능한 베이 확인 → 베이 선택 → 즉시 결제 → 세차 세션 시작 → 사용 중 → 세차 완료 → 사용 가능 복귀

### POC 핵심 목표
> 고객 앱에서 **사용 가능한 베이를 선택·즉시 결제**하면 관리자 화면에서 해당 베이가 **사용 중**으로 바뀌고, **세차 완료** 시 앱과 관리자 양쪽에서 다시 **사용 가능**으로 돌아오는 흐름을 구현한다. → **핵심은 앱↔관리자 간 베이 상태 동기화 하나입니다.**

### 이번 POC에서 실제 구현할 기능 (이 7개만)
1. 베이 상태를 서버·DB에 저장·조회
2. 고객 앱과 관리자 화면이 동일한 베이 데이터를 사용
3. 앱에서 결제 완료 상태 생성
4. 세차 세션의 **시작 시각·종료 예정 시각** 저장
5. 결제 완료 시 베이를 **사용 중**으로 변경
6. 앱 또는 관리자에서 완료 처리 시 베이를 **사용 가능**으로 변경
7. 동일 베이 **중복 사용 방지** 및 **POC 데이터 초기화**

### 간소 구현
- 앱 로그인: 테스트 계정 1~2개 · 관리자 로그인: 단일 관리자 계정
- 결제: 실제 PG 없이 **결제 완료 상태만** 생성
- 실시간 반영: WebSocket 대신 **3~5초 주기 조회** 가능
- 권한: 일반 사용자 / 관리자 2종만
- 차량 정보: 테스트 계정에 **미리 등록된** 차량 정보 사용

### 화면 모의 유지 (기존 화면·클릭 반응만 유지, 서버 처리 없음)
- 비전 AI 모니터링 · 대시보드 KPI/차트 · 정산 통계 · 실시간 알림 · 상세 매출 분석

### POC 제외
- 실제 소셜 로그인 · 실제 PG 결제 · SMS/카카오 알림 · 실제 비전 AI·번호판 인식
- 다지점 관리 · 다층(2F) 베이 · 복잡한 권한 관리 · 상세 감사 로그 · 대규모 트래픽 대응

### POC 완료 기준
> 앱에서 베이 선택·결제 → 관리자 **사용 중** 반영 → 앱 또는 관리자에서 세차 완료 → 앱·관리자 모두 **사용 가능** 복귀, 이 흐름이 개발자의 별도 조작 없이 **3회 연속 정상 동작**한다.

## 1. 문서 목적

이 문서는 영차 POC 데모의 **프론트엔드 화면 기준**으로 백엔드 구현 범위와 POC 간소화 범위를 확정합니다. 완성형 설계서가 아니라 ①사용자가 무엇을 할 수 있어야 하는가 ②행동 후 어떤 결과가 나와야 하는가 ③무엇을 완료로 인정하는가 ④잘못 구현됐을 때 무엇이 문제인가 를 판단하기 위한 문서이며, 이 문서 하나로 구현 범위와 검수가 가능하도록 구성했습니다.

## 2. POC 목표

**영차(Young차)** 는 고객이 앱에서 세차장 베이를 이용하는 **세차장 베이 이용 서비스**입니다. 고객은 **현재 사용 가능한 베이를 실시간으로 확인하고 즉시 결제한 뒤, 결제한 베이에서 바로 세차**하고 시간 연장·완료까지 처리합니다. 관리자는 베이 사용 현황을 관리합니다.

> **서비스 정책**: 영차는 **베이 예약 서비스가 아닙니다.** 미래 시간·특정 시간대의 베이를 미리 예약하는 기능은 제공하지 않으며, 베이 이용은 **사용 가능한 베이의 즉시 결제**로만 시작됩니다. 예약 등록·변경·취소·대기·예약 상태는 개발하지 않습니다.

**증명할 핵심 가치 (한 문장):** 앱 즉시 결제 → 관리자 사용 중 반영 → 완료 → 사용 가능 복귀의 **베이 상태 동기화가 하나의 데이터로 일관되게** 동작함을 증명한다.

> ⚠️ **현재 데모의 공백**: 앱(`yc_mobileapp`)과 관리자(`yc_pc_admin`)는 **서로 분리**되어 데이터를 공유하지 않습니다. 양쪽 베이 상태가 각각 하드코딩되어 서로 반영되지 않습니다. **이 연결이 POC의 핵심 구현 목표**입니다.

## 4. 핵심 POC 시나리오

### 시나리오. 빈 베이 선택 → 결제 → 세차 → 완료 (앱 ↔ 관리자 동기화)
1. **[앱]** 로그인 후 홈에서 베이 현황 확인 (상태: 사용 가능/사용 중/점검 중)
2. **[앱]** '사용 가능' 베이 탭 → 결제 바텀시트에서 세차 상품 선택 (예: 기본 세차 ₩8,000)
3. **[앱]** '결제하기' → 결제 완료 → 세차 진행 화면 이동
4. **[동기화]** 해당 베이가 관리자 화면에서 **사용 중**으로 변경 `← 핵심 구현`
5. **[앱]** 세차 화면 타이머 동작, 필요 시 시간 연장 (종료 예정 시각 갱신)
6. **[앱]** 타이머 종료 → 완료 처리
7. **[동기화]** 해당 베이가 관리자 화면에서 다시 **사용 가능** `← 핵심 구현`. 관리자 '이용 종료'로도 동일 결과, 앱에 반영
8. **[검수]** 앱·관리자 양쪽 **새로고침 후에도** 동일 상태 유지

### 예약 관련 UI 삭제 대상 (프론트엔드 수정)

> 현재 관리자 데모에 존재하는 '예약됨' 상태와 '예약 취소' UI는 서비스 정책과 맞지 않으므로 프론트엔드 수정 시 삭제 대상이다. 백엔드 기능과 데이터는 구현하지 않는다.

아래 항목은 단순 비활성화가 아니라 **화면에서 삭제**하는 것을 기준으로 한다.

- 관리자 화면의 '예약됨' 상태 표시
- '예약 취소' 버튼 또는 메뉴
- 상태 필터의 '예약됨' 항목
- 예약 관련 샘플 데이터
- 예약 관련 배지·문구·아이콘

> 현재 화면에서 '예약됨'으로 표시되는 베이는 화면 맥락에 맞게 **사용 가능** 또는 **점검 중**으로 변경한다. 예약 상태 자체는 남기지 않는다.

## 6. POC 최소 데이터

> 코드에 실제 존재하는 필드·표시값 우선. 완성형 ERD 아님.

**세차 베이 (Bay)** — *핵심*
- 베이 ID / 번호(1~6) / 상태(사용 가능·사용 중·점검 중) / 현재 연결된 세션 ID

**세차 세션 (Session)** — *핵심*
- 세션 ID / 베이 ID / 사용자 ID / 상품명·결제금액
- **시작 시각 / 종료 예정 시각** (남은 시간은 저장하지 않고 클라이언트가 계산 — 아래 8절)
- 연장 횟수·연장 금액 / 상태(진행 중·완료)

**결제 (Payment)** — *핵심(완료 상태만)*
- 주문번호 / 사용자 ID / 세션 ID / 금액 / 상태(결제 완료)

**세차 상품 (Service)** — *간소*
- 상품명(기본 세차 등) / 가격 (앱 표시 가격 기준, 예: 기본 세차 ₩8,000)

**고객 (Customer)** — *간소, 미리 등록*
- 사용자 ID / 이름 / 연락처 / 차량번호 / 차종·색상

> 결제 목록·환불·타임세일 요금표·매출 통계·비전 로그는 **저장 대상 아님**(샘플/화면 모의 유지).
>
> **만들지 않는 것 (예약 미제공)**: 예약 테이블 · 예약 API · 예약 상태 · 예약 일시 · 예약 취소 이력 · 사용자별 예약 목록. 베이 이용은 **결제 완료 시 즉시 세차 세션이 생성되는 방식으로만** 처리한다.

## 7. POC 상태 및 처리 규칙

### 베이 상태 — 3종만 사용
`사용 가능` · `사용 중` · `점검 중`
- `예약됨` 상태는 **사용하지 않는다**(서비스 정책상 예약 미제공). 현재 화면에서 '예약됨'으로 표시되는 베이는 화면 맥락에 맞게 **사용 가능** 또는 **점검 중**으로 변경한다.
- `수리 중` 상태는 사용하지 않으며 **점검 중으로 통합**한다.

### 상태 전이표
| 대상 | 현재 상태 | 변경 행동 | 다음 상태 | 함께 변경 |
|---|---|---|---|---|
| 베이 | 사용 가능 | 앱 결제 완료 | 사용 중 | 세션·결제 생성, 시작·종료예정 시각 |
| 베이 | 사용 중 | 앱 타이머 종료 / 관리자 이용 종료 | 사용 가능 | 세션 완료 |
| 세션 | 진행 중 | 앱 시간 연장 | 진행 중(유지) | **종료 예정 시각 변경**·연장요금 |
| 베이 | 사용 가능/사용 중 | 관리자 점검 등록 | 점검 중 | 세션 연결 해제 |
| 베이 | 점검 중 | 관리자 운영 재개 | 사용 가능 | - |

### 처리 규칙 (POC 확정)
- **즉시 결제 = 세차 시작**: 결제 완료 시 세션 생성 + 해당 베이 즉시 사용 시작(사용 중). 별도 '작업 시작' 버튼 없음. **사전 예약 단계 없음.**
- **완료 = 출차**: 앱 타이머 종료 또는 관리자 이용 종료를 완료로 처리, 베이 사용 가능. 별도 출차 상태 없음.
- **중복 사용 방지**: '사용 가능' 상태에서만 결제 성공. 동시 결제 시 하나만 성공(같은 베이에 두 세션 연결 불가).

## 8. 저장·동기화 기준

| 기능 | 새로고침 유지 | 재접속 유지 | 앱↔관리자 반영 | 반영 방식 |
|---|---|---|---|---|
| 베이 상태 | 필수 | 필수 | **필수** | 3~5초 주기 조회 |
| 세차 세션(시각) | 필수 | 필수 | 필수 | 주기 조회 |
| 결제 완료 | 필수 | 필수 | 관리자 반영 | 주기 조회 |
| 대시보드/결제목록/비전 | 불필요 | 불필요 | 불필요 | 화면 모의 |
| 테마/사이드바 | 유지 | 유지 | 불필요 | localStorage |

### 타이머 저장 기준 (중요)
- 서버에 **남은 시간을 매초 저장하지 않는다.**
- 서버에는 **시작 시각·종료 예정 시각**만 저장한다.
- 앱은 **현재 시각 vs 종료 예정 시각**을 비교해 남은 시간을 계산한다.
- 새로고침·재접속해도 **동일한 종료 예정 시각 기준**으로 타이머가 이어진다.
- 시간 연장은 **종료 예정 시각을 변경**하는 방식으로 처리한다.

### 일관성 기준
베이 상태의 **단일 소스(서버)** 를 두고 앱 베이맵·관리자 대시보드·베이관리가 모두 같은 값을 조회한다. **WebSocket은 POC 필수 아님**(주기 폴링으로 충분). 화면마다 다른 상태가 표시되면 안 된다.

## 11. POC 검수 체크리스트

- [ ] 세차 시작 시각·종료 예정 시각이 서버에 저장된다.
- [ ] 새로고침 후에도 타이머가 동일 종료 예정 시각으로 이어진다.
- [ ] 시간 연장 시 종료 예정 시각이 변경된다.
- [ ] 앱에서 완료하면 베이가 사용 가능으로 바뀐다.
- [ ] 관리자 이용 종료 시 베이가 사용 가능으로 바뀐다.
- [ ] 앱과 관리자의 베이 상태가 항상 일치한다.
- [ ] 서버 오류 시 화면 전체가 멈추지 않는다.
- [ ] 데모 초기화 기능이 동작한다.
- [ ] 핵심 흐름이 3회 연속 정상 동작한다.

**예약 미제공 확인**
- [ ] 앱과 관리자 화면 어디에도 예약 기능이나 예약 상태가 표시되지 않는다.

- [ ] 예약 등록·변경·취소 버튼이 존재하지 않는다.

## 12. 테스트 데이터 및 초기화 기준

### 코드에서 확인된 샘플
- **관리자 계정**: `admin / 1234`
- **앱 계정**: 없음(소셜 가짜) → 테스트 계정 1~2개 생성 필요, 차량 정보 미리 등록
- **베이**: 1차 POC는 **6개**. 초기 상태 예 — 일부 사용 가능, 1개 점검 중
- **고객·차량(관리자 드로어)**: 김영수 / `12가 3456` / 현대 쏘나타 DN8 / 화이트 등
- **상품·가격**: 앱 기본 세차 ₩8,000 (앱 표시 가격 기준)

### 시작/완료 상태
- 시작: 일부 베이 사용 가능, 세션·결제 없음(또는 최소 샘플)
- 완료: 시나리오 1회 수행 후 결제·세션 1건, 베이 사용가능→사용중→사용가능 순환 확인

### 초기화 방법(제안)
- 개발 전용 **초기화 API**(예: `POST /dev/reset`) 또는 관리자 화면 초기화 버튼(개발 모드) 또는 시드 재입력 스크립트

## 13. POC 확정 사항

**서비스 정책 — 예약 미제공 (대표이사 확정)**
- 영차는 **베이 예약 서비스가 아니다.** 고객은 사용 가능한 베이를 확인하고 **즉시 결제 후 바로 이용**한다.
- 미래 시간·특정 시간대 베이의 사전 예약 기능은 제공하지 않는다.
- 예약 등록·변경·취소·대기·예약 상태는 개발하지 않는다. 예약 관련 UI·상태·데이터·API는 모두 제거 대상이다.

**앱·관리자 데이터**
- 고객 앱과 관리자는 **동일 서버·DB** 사용. 베이 상태는 **서버 데이터를 단일 기준**으로 사용. 화면별 하드코딩 베이 상태는 백엔드 연결 후 사용하지 않는다.

**베이 개수**
- 1차 POC는 앱·관리자 모두 **6개** 베이만 사용. 관리자 7~12번은 화면에서 숨기거나 POC 제외 처리. 2층은 제외.

**베이 상태**
- **사용 가능 · 사용 중 · 점검 중** 3종만 사용. `예약됨`은 사용하지 않으며(사용 가능 또는 점검 중으로 변경), `수리 중`은 점검 중으로 통합.

**결제와 세차 시작**
- 사용 가능 베이의 **즉시 결제**로만 이용 시작. 결제 완료 시 세션 생성 + 베이 사용 중. 별도 작업 시작 버튼 없음. 사전 예약 단계 없음. 실 PG 미연결.

**세차 완료**
- 앱 타이머 종료 또는 관리자 이용 종료를 완료로 처리, 베이 사용 가능. 별도 출차 상태 없음.

**차량 정보**
- 앱 테스트 계정에 차량번호·정보 미리 등록. 차량 등록·수정 화면 미개발. 비전 AI 인식은 화면 모의.

**관리자 계정**
- 관리자 로그인만 간소 구현. 계정 등록·수정·삭제는 없으며 POC 제외.

**결제 가격**
- 앱·관리자 가격이 다르지만 1차 POC는 **앱 결제 화면 표시 가격 기준**. 관리자 상품 가격 관리·타임세일 가격 연동은 제외.
